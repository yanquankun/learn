### 我把 OpenClaw 搬进了像素办公室：Pixel Office 搭建全记录

![image-20260312150952787](https://oss.yanquankun.cn/oss-cdn/img/image-20260312150952787.png!watermark)

如果你也觉得“让 AI 在终端里默默干活”虽然高效，但**观感像黑客片花絮**，那你大概会喜欢 Pixel Office 这种邪门又可爱的东西：

一群 AI，不再只是后台进程，而是在一个像素办公室里跑来跑去、开工摸鱼、执行任务、切换状态，甚至还能把日志直接贴在现场。

简单说：

- 以前你看的是：`agent started / tool call / task done`
- 现在你看的是：**Kai 正在电脑前敲键盘，Aria 站在白板旁边，Nova 刚从饮水机旁边路过**

这篇文章就讲清楚：

1. 我们为什么要做 Pixel Office
2. 它到底有什么用
3. OpenClaw 和 VPS 需要怎么配
4. `openclaw-pixel-office` 仓库怎么接入
5. 中间踩了哪些坑
6. 未来还能扩展成什么离谱但有用的东西

---

## 一、为什么要做这个

### 1. 因为多 Agent 系统太会“隐身办公”了

OpenClaw 跑起来之后，其实很强：

- 可以多 Agent 协作
- 可以接 Discord / Feishu / Web
- 可以调浏览器、节点、消息、任务调度
- 可以后台持续跑任务

问题是：**它干活的时候，不够“可见”**。

很多时候你知道它在跑，但不知道：

- 现在是谁在干活？
- 它是在忙，还是卡死？
- 它刚刚调用了什么工具？
- 它到底是在建设项目，还是在进行一场像素世界的哲学漫步？

于是 Pixel Office 的目标就非常明确：

> 把抽象的 Agent 运行状态，变成一个可视化、可感知、可展示的“办公室现场”。

### 2. 因为“会动的状态面板”比一堆日志更像产品

纯日志当然有用，但日志通常是给开发者看的。

而 Pixel Office 更像是：

- 一个演示层
- 一个监控层
- 一个团队氛围层
- 一个“让老板一眼就知道系统还活着”的精神稳定器

说人话就是：

**同样是 Agent 在工作，像素办公室比黑底白字更容易让人相信这玩意真的在上班。**

---

## 二、Pixel Office 有什么作用

当前这套系统，至少有 5 个实际用途。

### 1. 实时展示 Agent 工作状态

比如：

- 空闲
- 执行工具
- 任务完成
- 正在跟随某个工作流推进

当 watcher 监听到 OpenClaw 的 session JSONL 变化后，可以把行为映射到像素角色的动作或状态展示上。

### 2. 让多 Agent 协作“看得见”

比起“现在有 pm / dev / ui / qa / ops / data 六个机器人在工作”，
更直观的方式是：

- PM 在中间区域调度
- Dev 在工位前埋头敲代码
- UI 在展示墙附近工作
- Ops 在设备区附近出没

当视觉映射合理以后，团队协作会一下子从“逻辑存在”变成“视觉存在”。

### 3. 做为 OpenClaw 的展示界面

如果你要给团队内部、客户、朋友、投资人、或者隔壁工位那个总觉得 AI 只会胡说八道的同事演示，Pixel Office 比控制台友好得多。

它有一种天然的产品感：

> “这不是一堆脚本，这是一个正在运行的 AI 办公空间。”

### 4. 作为任务活动日志的前台窗口

我们在这次接入里，把日志链路明确成了：

`sessionDir -> watcher 解析 JSONL -> activity log file -> /api/activity-logs -> 前端日志面板`

这样前台日志展示就不是拍脑袋乱拼，而是有一条清晰的数据来源路径。

### 5. 未来可以变成“可交互的 Agent 总部”

别看现在只是个办公室，往后很容易长成：

- Agent 任务墙
- 工单调度屏
- 语音播报角
- 群消息指挥台
- 远程节点状态室
- 机器人“出勤监控大屏”

说白了，像素办公室只是皮肤；真正有价值的是它背后的 Agent 编排与状态映射。

## 三、系统架构：它到底怎么串起来的

可以把它理解成四层：

### 第 1 层：OpenClaw Agent 运行层

OpenClaw 负责：

- 接收消息
- 分派任务
- 运行 Agent
- 调用工具
- 记录 session transcript

这些 session transcript 一般在：

```bash
~/.openclaw/agents/<agent-id>/sessions/*.jsonl
```

### 第 2 层：Watcher 监听层

Pixel Office 服务端会监听某个 session 目录，比如：

```bash
/home/<USER>/.openclaw/agents
```

监听到 JSONL 更新后：

- 解析 toolCall / 行为事件
- 映射成 Agent 动作
- 写入活动日志文件

### 第 3 层：Pixel Office 服务端

服务端负责：

- 读取配置
- 提供 `/api/config`
- 提供 `/api/activity-logs`
- 提供 WebSocket 推送状态
- 加载像素资产和办公室布局

### 第 4 层：Web 前端

前端负责：

- 渲染办公室
- 显示像素角色
- 显示日志弹层
- 显示配置页
- 做交互（比如选中角色、关闭名牌、查看日志等）

## 四、我们这次实际做了什么

这次我们做的不是“修个按钮就下班”，而是一整套升级。

### 1. 配置页去掉无效项

我们先排查了配置页里哪些东西真的生效。

最终确认：

- `OpenClaw 启用`：有用
- `Session 目录`：有用
- `API 端点`：当前未接入，没用
- `飞书机器人 Webhook`：当前没接逻辑，没用

所以把没接入的内容删掉了。

这是一个很朴素但很重要的原则：

> 配置页不是许愿池，别把没接线的按钮放在那里假装高科技。

### 2. 新增活动日志输出路径配置

这次新增了一个明确配置项：

- `activityLogPath`

例如：

```json
{
  "openclaw": {
    "enabled": true,
    "sessionDir": "/home/<USER>/.openclaw/agents",
    "activityLogPath": "/home/<USER>/.pixel-agents/activity.log"
  }
}
```

这件事很关键，因为它把“日志从哪来”和“日志展示读哪”分开了：

- `sessionDir` = 输入源
- `activityLogPath` = 输出展示源

### 3. 修了名牌关闭按钮

之前的关闭按钮点击没反应，原因非常朴实：

- `onCloseAgent` 是空函数

也就是说，按钮的职责是“负责看起来像按钮”。

现在已经改成：

- 点击后取消选中角色
- 如果正在 camera follow，也会解除

### 4. 办公室从“能用”升级为“高拟真版”

![image-20260312151036692](https://oss.yanquankun.cn/oss-cdn/img/image-20260312151036692.png!watermark)

一开始的办公室更像基础装修：

- 桌子
- 椅子
- 书架
- 绿植
- 饮水机

后来继续升级为真实 tileset 家具版：

- 窗户
- 墙面挂件 / 图表 / 白板
- 显示器 / CRT / Laptop
- 打印机 / 电话 / 服务器
- 咖啡机 / 咖啡杯
- 贩卖机 / 冰箱 / 储物柜
- 多种座椅 / 小型桌面装饰

最终服务端确认加载：

```text
[AssetLoader] Loaded 92/92 furniture assets
```

这时候它才终于从“像素办公室概念图”进化成“像素办公室真装修”。

## 五、OpenClaw 侧配置示例

下面给一个**脱敏版示例**。

> 注意：只展示和本项目接入相关的关键部分，不代表你的完整 openclaw.json 必须长这样。

```js
{
  gateway: {
    port: <OPENCLAW_GATEWAY_PORT>,
    mode: 'local',
    bind: 'loopback',
    auth: {
      mode: 'token',
      token: '<YOUR_OPENCLAW_TOKEN>'
    },
    controlUi: {
      allowedOrigins: ['https://<YOUR_DOMAIN>']
    }
  },

  web: {
    enabled: true
  },

  commands: {
    native: true,
    nativeSkills: true,
    text: true,
    bash: true,
    config: true,
    debug: true,
    restart: true
  },

  channels: {
    discord: {
      enabled: true,
      commands: {
        native: true,
        nativeSkills: true
      },
      accounts: {
        default: {
          token: '<YOUR_DISCORD_BOT_TOKEN>',
          commands: {
            native: true,
            nativeSkills: true
          }
        }
      }
    }
  }
}
```

### 一个非常容易踩的坑：全局开了，不代表账号级没关

我们这次就踩到了这个坑。

虽然全局是：

```js
channels.discord.commands.native = true
```

但账号级配置居然是：

```js
channels.discord.accounts.<account>.commands.native = false
```

效果就是：

> 总闸开着，分闸全关了。

然后你就会怀疑人生：

“咦？以前那个内置命令弹出是不是版本没了？”

其实不是，它只是被账号级配置给温柔地暗杀了。

## 六、Pixel Office 服务侧配置

我们这边的项目配置重点是：

```json
{
  "version": 1,
  "agents": [
    { "id": 1, "name": "PM", "role": "pm" },
    { "id": 2, "name": "Dev", "role": "dev" },
    { "id": 3, "name": "UI", "role": "ui" }
  ],
  "openclaw": {
    "enabled": true,
    "sessionDir": "/home/<USER>/.openclaw/agents",
    "activityLogPath": "/home/<USER>/.pixel-agents/activity.log"
  }
}
```

### 配置项说明

- `enabled`
  - 是否启用 OpenClaw watcher
- `sessionDir`
  - OpenClaw session transcript 根目录
- `activityLogPath`
  - 给前端日志面板读取的活动日志文件

## 七、VPS 需要配置什么

### 1. Node 环境

至少你需要：

- Node.js 20+
- npm
- systemd user service（推荐）
- Nginx 反向代理（推荐）

### 2. systemd 用户服务

像这样：

```ini
[Unit]
Description=OPC Pixel Office
After=network.target

[Service]
Type=simple
WorkingDirectory=/home/<USER>/projects/openclaw-pixel-office
ExecStart=/usr/bin/node /home/<USER>/projects/openclaw-pixel-office/dist/server/server/index.js
Restart=always
RestartSec=3
Environment=NODE_ENV=production

[Install]
WantedBy=default.target
```

然后：

```bash
systemctl --user daemon-reload
systemctl --user enable --now opc-pixel-office.service
systemctl --user status opc-pixel-office.service
```

### 3. Nginx 反向代理

例如：

```nginx
location /opc-pixel/ {
    proxy_pass http://127.0.0.1:3214/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

如果你前端和 WebSocket 都走这个服务，记得把升级头带上，不然 WebSocket 会表演当场去世。

## 八、仓库接入指南

仓库地址：

<https://github.com/yanquankun/openclaw-pixel-office>

### 1. 拉代码

```bash
git clone git@github.com:yanquankun/openclaw-pixel-office.git
cd openclaw-pixel-office
npm install
```

### 2. 构建

```bash
npm run build
```

### 3. 准备运行期配置

在运行用户目录准备配置，例如：

```bash
mkdir -p /home/<USER>/.pixel-agents
```

然后生成项目配置文件（按项目实际要求放置）：

```json
{
  "version": 1,
  "agents": [
    { "id": 1, "name": "PM", "role": "pm" },
    { "id": 2, "name": "Dev", "role": "dev" },
    { "id": 3, "name": "UI", "role": "ui" },
    { "id": 4, "name": "QA", "role": "qa" },
    { "id": 5, "name": "Ops", "role": "ops" },
    { "id": 6, "name": "Data", "role": "data" }
  ],
  "openclaw": {
    "enabled": true,
    "sessionDir": "/home/<USER>/.openclaw/agents",
    "activityLogPath": "/home/<USER>/.pixel-agents/activity.log"
  }
}
```

### 4. 确保 OpenClaw 在跑

如果 OpenClaw 没在跑，Pixel Office 也不是完全不能开，但它就会像一个装修很好的空办公室。

你可以：

```bash
openclaw doctor --non-interactive
```

检查：

- Discord 是否正常
- Feishu 是否正常
- 会话目录是否存在
- 插件是否加载成功

### 5. 启动 Pixel Office 服务

```bash
npm run start
```

或者 systemd 启动：

```bash
systemctl --user restart opc-pixel-office.service
```

## 九、日志是怎么接入的

![image-20260312151308070](https://oss.yanquankun.cn/oss-cdn/img/image-20260312151308070.png!watermark)

这是很多人第一次看项目时会搞混的点。

### 不是“前端直接读 sessionDir”

前端不是直接拿 `.jsonl` 原文来显示。

### 正确链路是

```text
OpenClaw sessionDir
  -> watcher 监听 JSONL
  -> 解析 toolCall / 行为事件
  -> 写入 activityLogPath
  -> 服务端 /api/activity-logs
  -> 前端日志面板显示
```

这样做的好处：

1. 前端不用理解复杂 transcript 格式
2. 展示数据可控
3. 可以自由改日志文案风格
4. 后续可以接入更多来源，而不用把前端搞成 JSONL 考古现场

## 十、这次踩过的坑

### 坑 1：配置页出现“没接线的按钮”

表现：

- 看起来什么都能配
- 实际有些字段根本没被用到

处理方式：

- 把无效项删掉
- 只保留真实生效项
- 对未来功能再单独接入，不提前假装存在

### 坑 2：Discord 指令弹出消失

表现：

- 以前输入指令会弹 OpenClaw 内置命令
- 现在没了
- 很容易怀疑是版本升级把功能干掉了

实际根因：

- 全局配置开着
- 账号级配置把 native commands 关了

处理方式：

- 检查 `channels.discord.commands`
- 再检查 `channels.discord.accounts.*.commands`

### 坑 3：按钮“有 UI 没逻辑”

这个非常经典。

按钮长得像按钮，不代表它具备按钮的法律效力。

我们这次的关闭按钮就是典型案例：

- `onCloseAgent` 是空函数
- 点它就像对空气发出管理请求

解决：

- 正式把关闭逻辑补齐
- 取消选中 / 解除跟随状态

### 坑 4：高拟真版不是摆几个桌子就算装修

你以为“多放几张桌子和绿植”就叫办公室升级。

不。

那叫“公司刚搬家，行政预算还没批下来”。

真正的高拟真，需要：

- 真实 tileset 家具资源
- 家具 catalog
- 合理的布局分区
- 墙面装饰、设备区、协作区
- 桌面小物件细节

最后这次我们把 92 个家具资产真正接进来了，效果才终于像回事。

### 坑 5：磁盘空间会被“不是主角的东西”吃掉

这次磁盘高占用，主要不是代码本身，而是：

- `/var/tmp` 里的 Homebrew 临时解包垃圾
- `journald` 历史日志
- npm 缓存
- 少量旧 session transcript

所以清理原则要记住：

> 优先删可再生缓存，最后才考虑业务数据。

不要一上来就对 session 目录挥舞删除大刀，像在数据中心 cosplay 狂战士。

## 十一、我们这次怎么安全清理磁盘

采用的是“低风险优先”原则：

### 清理对象

- `/var/tmp/homebrew-*`
- journald 归档日志
- npm cache
- node / pip / Homebrew 的可再生缓存
- 老旧 cron-worker session
- orphan transcript（不在 sessions.json 中引用的 transcript）

### 明确不碰

- 项目源码
- 当前运行服务
- 当前活跃 session
- 数据库
- 正在被引用的配置和状态文件

这类清理很适合做成定期维护 SOP。

## 十二、未来还能做什么

这部分其实最有意思。

### 1. 更真实的 Agent 状态动画

比如：

- 写代码时坐下
- 调用工具时走向设备区
- 等待时靠墙发呆
- 完成任务时弹出短提示

让状态从“图标变化”变成“行为变化”。

### 2. 任务墙 / 工单墙

在办公室里加一个大屏：

- 当前任务队列
- 阻塞项
- 负责人
- ETA

这会让它从展示页面进一步变成调度中台。

### 3. Discord / Feishu 双向联动

例如：

- Discord 发命令
- 像素办公室里角色立刻开始行动
- 前端看到状态变化
- 完成后再把结果回推到消息渠道

这会很像一个“可以看见执行过程”的机器人控制台。

### 4. 节点 / 浏览器 / 远程设备可视化

未来完全可以扩展出：

- Mac 节点房
- 浏览器任务室
- 语音呼叫角
- 摄像头监控屏

把 OpenClaw 的工具生态变成视觉化空间布局。

### 5. 大屏模式

如果你有办公室电视或者开发大屏，Pixel Office 很适合直接变成：

- 团队 AI 值班看板
- 演示墙
- 状态面板
- 直播型开发背景板

让“AI 正在工作”这件事第一次长得像一家公司，而不是一个隐藏在 tmux 里的秘密宗教组织。

## 十三、结语

Pixel Office 本质上不是为了可爱而可爱。

它真正的价值在于：

> 把看不见的多 Agent 工作流，变成看得见、能解释、能展示、能运营的系统界面。

而且这种可视化不是花架子。

当系统开始变复杂，参与的人开始变多，任务开始常驻后台之后，**“可见性”本身就是生产力**。

你可以把它理解成：

- OpenClaw 是大脑
- tools 是手脚
- Discord / Feishu 是嘴
- Pixel Office 是脸

一个系统如果只有大脑没有脸，确实也能活。

但有了脸之后，别人终于知道它不是一团在服务器里神神秘秘蠕动的算力。

它是在上班。

而且上得还挺认真。
